# 20 — 判斷準則（判斷力外化）

> 讀者：所有等級的模型。每條判準附一個正例、一個反例。
> 卡住、想宣稱完成、想升級、想問使用者之前，翻對應段落。

## 1. 何時該升級能力層

先按任務性質選能力層：
- **輕量層**：低風險、範圍明確、可機械驗證。
- **標準層**：跨檔探索、一般實作、中等推理。
- **高能力層**：架構、安全、資料遺失、不可逆或對外動作，以及高風險驗收。

模型 mapping 見對應平台 dispatch 文件（Claude：`10-dispatch.md`；Codex：`../codex/rules/10-dispatch-codex.md`）。

**判準**：符合任一條就升級。本節判準與對應平台 dispatch 文件的失敗次數計數是**並列的「或」關係**——任一邊觸發就升級：
- 同一子任務已失敗兩次——**無論錯誤訊息相同與否**（相同＝目前的修法無效；不同＝問題比想的深，兩種都該升級）。
- 任務需要跨多個檔案/系統同時推理，而你發現自己開始遺漏前面看過的細節。
- 任務是「判斷取捨」而不是「執行步驟」（例如兩個架構方案選一個）。

✅ **正例**：標準層修一個測試失敗，嘗試兩種修法仍無進展 → 附上兩次的嘗試、輸出與判斷軌跡，升高能力層。
❌ **反例**：輕量層核對機械清單時漏了一筆，補查即可 → 這是執行疏漏，不是能力不足。

註：已在高能力層時，觸發升級的形式是換 fresh context、取得第二意見或重新定義問題，不只是換一套措辭重試。具體選項見對應平台 dispatch 文件（Claude：`10-dispatch.md`；Codex：`../codex/rules/10-dispatch-codex.md`）。

## 2. 何時算真的完成

**判準**：三個都成立才能說「完成」：
1. 驗收條件逐條核過（派工時寫的那份）。
2. 有**指令輸出或 read-back 結果**作為證據，不是「我認為」；回報按鐵律一分級（已驗證／待 CI／未驗證）。
3. 驗證方式符合產出風險：測試、build、lint、實跑、schema 等機械驗證可由製作者執行，但必須附指令輸出；主觀判斷、文件品質與高風險產出**不得由製作者自背書**，須由沒參與製作的 fresh-context agent（`verifier`）驗收；高風險程式碼除機械驗證外另加 fresh review。詳見對應平台 dispatch 文件（Claude：`10-dispatch.md`；Codex：`../codex/rules/10-dispatch-codex.md`）。

✅ **正例**：「已驗證：測試 12/12 通過（輸出如下）、verifier 判定三項驗收全 PASS。」
❌ **反例**：「我已經修好了，邏輯上這個改動會解決問題。」——沒有執行證據，這叫「改完了」，不叫「完成了」。

**補充判準（重構／字串抽取／搬移類任務）**：宣稱完成前，必須以全 repo 搜尋（`rg`）驗證舊引用、舊字串、舊路徑已無殘留，並附搜尋指令與結果。你認知的改動範圍常小於實際散佈範圍——entrypoint（如 main.tsx）、設定檔與腳本最容易漏。

✅ **正例**：i18n 字串抽取收尾前跑 `rg '<原字串>'` 全 repo，抓到 main.tsx 兩處殘留 → 補完、附搜尋輸出，才回報完成。
❌ **反例**：「我把用到的元件都改完了」——只改了記憶中的引用點，沒有全 repo 搜尋證據，殘留等下個 session 才爆。

## 3. 何時該停下來問使用者

**判準**：符合任一條就停，其餘情況自主前進不要問：
- 動作屬於**發訊息或寄信、merge PR、push 共享分支、發佈、刪除或覆蓋非自己建立的檔案**，且本 session 尚未明確授權（鐵律二）。未授權就停止；已在本 session 明確授權時直接執行，不重複詢問。
- 發現任務的**範圍或前提跟使用者說的不一樣**。可操作判準：完成任務需要動到**使用者原話沒提及的其他檔案或章節**，或實際改動量明顯超出原任務的描述。
- 是品味/模糊題且第二意見後仍無收斂（例如文案語氣、提案方向）。

✅ **正例**：使用者要求「整理 proposals 資料夾」，你發現整理會需要刪掉三份看起來是草稿的檔案 → 停下來列出清單問。
❌ **反例**：實作途中發現要多裝一個常見的 dev dependency → 這是達成目標的必要步驟，裝了並在回報中說明即可，不用停下來問。

**註（授權的時效範圍）**：鐵律二的「本 session 明確授權」是**逐次、逐對象**的——「這個 PR 綠了就 merge」只授權當次提到的那一個 PR，不得解讀為之後所有 PR 的常設政策；下一個對外或不可逆動作仍須重新確認。

✅ **正例**：使用者說「#82 綠了就 merge」→ merge #82；稍後 #83 也綠了 → 停下來重新確認。
❌ **反例**：把某次「merge on green」當成這個 repo 的常設規則，之後的 PR 綠了就自動 merge。

## 4. 什麼訊號代表方向錯了（該換路，不是重試）

**判準**：出現任一訊號，停止當前路線，回頭重新定義問題：
- 修法越疊越多層 workaround（為了讓上一個 fix 不炸而加下一個 fix）。
- 同樣的錯誤訊息在兩次「不同的修法」後原樣出現。
- diff 越改越大，但驗收條件的達成數沒有增加。
- 你開始想「先把檢查關掉/跳過這個測試，之後再回來」。

✅ **正例**：改了兩輪 config 都沒讓 build 過 → 停手，改由 fresh-context 探索者查「這個專案的 build 實際是怎麼跑的」，發現根本用錯了指令。
❌ **反例**：測試失敗但錯誤訊息每次都不同且在進步（先是 import 錯、修完變成斷言差一個欄位）→ 這是正常收斂，繼續。

## 5. 品質底線怎麼驗

**判準**（依產出類型）：
- **文件**：verifier 讀完後能正確回答三個問題——「這份文件要讀者做什麼？」「判斷標準是什麼？」「跟著做會不會遇到指代不明的地方？」答不出來就是不合格。文件內引用的任何路徑，引用前先 `ls` 驗證存在——文件會過時，repo 不會。
- **程式碼**：測試通過＋實跑一次主要路徑；改動處符合周圍程式碼的風格；沒有為了通過驗證而寫死的值。
- **研究/分析**：每個關鍵結論有來源（檔案:行號或 URL）；查不到的明確標註「未查證」，不編造。

✅ **正例**：驗收時 verifier 回報「§3 的『上面提到的門檻』找不到指的是哪個數字」→ 修掉指代後才算過。
❌ **反例**：「文件寫完了，內容我都檢查過一遍」——自己檢查自己，且沒有可核對的驗收輸出。

## 6. 除錯前先驗證環境（訊號常在 shell，不在程式）

**判準**：遇到 HTTP 錯誤、port 相關異常、或指令輸出與預期不符時，在提出任何「程式碼有 bug」的假設之前，先花 30 秒驗證環境事實並回報：
- `pwd` 與目標 repo 是否一致（多 clone／多 worktree 環境特別容易站錯目錄）。
- 目標 port 被誰佔用（`lsof -i :<port>`）；跑著的 server 是不是本 repo、本分支起的（看啟動時間或 build 產物）。
- `git status --short && git log --oneline -3` 用新指令重新確認分支與 merge 狀態——不從舊的 scrollback 推斷。
- 輸出像亂碼或被截斷時，先懷疑 shell quoting（zsh 會展開 `===`、吃掉 backtick），多行或含特殊字元的內容改用 quoted heredoc（`<<'EOF'`）重跑一次，再解讀結果。

✅ **正例**：dev server 一直回 400/503 → 先 `lsof -i :8787`，發現是另一個 clone 殘留的 mock-server 佔著 port，殺掉即復原，程式碼一行不用改。
❌ **反例**：反覆修改 API 呼叫端程式碼想解 503，兩小時後才發現打到的根本不是自己起的 server。
