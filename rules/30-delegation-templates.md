# 30 — 派工 prompt 模板

> 讀者：主對話。派 subagent 時複製對應模板，把【】填滿。
> 【】欄位是必填；填不出「驗收條件」代表你還沒想清楚要什麼，先想再派。

## 通用規則（每個模板都適用）

- Subagent 看不到主對話，**所有必要脈絡都要寫進 prompt**（相關檔案路徑、使用者的原話、已知的限制）。
- 給 subagent 的路徑一律寫**絕對路徑**——它的工作目錄認知可能跟你不同，相對路徑會找錯地方。
- 回報上限 30 行；長產物落檔傳路徑。回報須分級：已驗證／待 CI／未驗證。
- model 選擇見 `10-dispatch.md` §1 的表。
- prompt 開頭明寫：「**你是被派來的執行者，親自完成本任務，不要再呼叫 Agent 工具轉包**」——subagent 也會讀到全域 rules，不加這句會把「指揮官不下場」套在自己身上、逐層轉包（2026-07-07 實測發生過 5 層遞迴）。收到「我已再派背景工作」類回報＝未完成，立即糾正。

## A. 搜尋／掃描（subagent_type: Explore）

```
目標：找出【要找什麼，例如：所有提到 Firebase Dynamic Links 的文件】。
動機：【為什麼要找，例如：要評估遷移影響範圍】。
範圍：【絕對路徑；要不要包含歷史/草稿】。
驗收條件：每一筆附 檔案:行號 與一行摘要；若超過 20 筆，改回報「總數＋分佈統計＋前 10 筆」。
回報格式：條列，每筆一行。找不到就明說「0 筆」，並列出你搜過的關鍵字組合。
```

## B. 實作（subagent_type: general-purpose）

```
目標：【要做出什麼行為，例如：在 X 頁面加上 Y 功能】。
動機：【使用者的原始需求一句話】。
脈絡：相關檔案在【絕對路徑】；本專案慣例是【stack/風格，例如 Vue 3 + TS，跟隨現有元件寫法】。
限制：最小必要變更；不改【不准動的範圍】；不新增依賴除非必要（新增了要在回報中說明）。
驗收條件：【可執行的驗證，例如：npm test 通過／實跑 curl X 回傳 Y】。做完必須實際執行驗證，以輸出為證。
回報格式：改了哪些檔（檔案:行號）、驗證指令與輸出關鍵行、任何偏離本 prompt 的決定、回報分級。
```

## C. 重構（subagent_type: general-purpose）

```
目標：把【範圍】重構為【目標狀態】，行為不得改變。
動機：【為什麼值得重構】。
驗收條件：(1) 重構前先跑一次【測試指令】記錄基準；(2) 重構後同一指令結果一致；(3) 對外介面／輸出格式不變。
安全規則：一次一小步；每步可獨立驗證；發現需要「順便修 bug」時停下來回報，不要混在重構裡做。
回報格式：改動摘要（檔案:行號）、前後測試輸出對照、發現但沒動的問題清單。
```

## D. 研究／查證（subagent_type: general-purpose）

```
目標：回答【具體問題】。
動機：【這個答案會拿來做什麼決定】。
方法：優先查【官方文件／repo 內既有文件】；每個結論附來源（URL 或 檔案:行號）。
誠實規則：查不到就標「未查證」，不得以訓練記憶充當查證結果；有多種說法時並列並註明各自來源。
驗收條件：問題的每個子項都有「答案＋來源」或「未查證＋查過哪裡」。
回報格式：結論先行（3 行內），然後逐項列證據。全文落檔到【絕對路徑】，只回報路徑＋結論。
```

## E. 驗收（subagent_type: verifier）

> verifier 的角色設定（找碴不背書、PASS/FAIL 判定、額外檢查矛盾與路徑真實性）已內建在 agent 定義裡，
> prompt 只需要給素材：

```
產出檔案：【絕對路徑清單】。
驗收條件（逐條判 PASS/FAIL）：
1.【條件一，須可機械判定】
2.【條件二】
3.【條件三】
額外脈絡：【這份產出的原始需求一句話；有不准動的範圍也寫在這】。
```

需要更廣的對抗審查（不只驗收清單，而是「找出所有規則衝突、誤讀風險」）時，仍用 verifier，
把驗收條件寫成審查面向即可。
