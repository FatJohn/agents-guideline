# 10C — Codex 模型調度守則

> 讀者：Codex 主對話（指揮官）。每次要派工、選模型或驗收之前照著做。
> 本檔是 `rules/10-dispatch.md` 的 Codex 版；Claude Code 專用的 Agent 工具參數不要套到這裡。

## 0. Codex 可用角色（以當前 session 實際工具為準）

Codex 會在可用時提供 multi-agent / subagent 能力。常見角色：

| 角色 | 用途定位 |
|------|----------|
| `explorer` | 唯讀探索、掃 repo、找檔案、整理證據 |
| `worker` | 實作、修 bug、批次改檔 |
| `default` | 一般用途 fallback |
| `verifier` | fresh-context 驗收（本系統提供，安裝於 `~/.codex/agents/verifier.toml`） |

模型與 reasoning effort 用 Codex 語彙描述：`model`、`model_reasoning_effort`。若不指定，Codex 會依任務選擇；高風險判斷或驗收可指定 high effort，機械性探索可用較快模型或較低 effort。

## 1. 指揮官不下場

以下工作應派 subagent 或保持在獨立脈絡中，主對話只收結論。判斷基準：會把超過一頁原始內容帶進主對話的動作，都不該自己做。

| 工作 | 建議角色 | 備註 |
|------|----------|------|
| 掃 repo、找出「哪些檔案有 X」 | `explorer` | 回報檔案:行號與摘要，不貼大量原文 |
| 讀多份長文件並總結 | `explorer` 或 `worker` | 長產物落檔，只回重點 |
| 查網頁、抓文件 | `worker` | 優先官方/原始來源，附 URL |
| 批次機械性改檔 | `worker` | 寫入前確認不和其他 agent 衝突 |
| 實作一個功能 | `worker` | 需求與驗收條件要寫完整 |
| 高風險判斷或架構取捨 | 高 effort agent 或第二意見 | 分歧點交回主對話決策 |
| 驗收任何檔案產出 | `verifier` | 找碴，不背書 |

主對話可以自己做的：讀單一已知檔案的特定段落、改一兩個檔、跑一條指令、跟使用者對話、做決策。

## 2. 派工三件套

每個 subagent prompt 必含三段，缺一段就是不合格派工（模板見 `30-delegation-templates-codex.md`）：

1. **目標與動機**：要達成什麼、為什麼；subagent 看不到完整主對話，脈絡要自帶。
2. **驗收條件**：可機械判定的完成定義；判準是另一個 agent 能只憑這句話判定過或不過。
3. **回報格式**：規定回什麼、多長、哪些證據必要。

## 3. 回報合約

- Subagent 只回**結論與證據**（檔案:行號、指令輸出關鍵行、URL），不回原始內容傾倒。
- 長產物（報告、大 diff、清單）落檔傳路徑；臨時的放 scratchpad，要留的放 repo 內合適路徑。
- 回報上限預設 30 行；需要更多就落檔。
- 回報必須分級：**已驗證（附證據）／待 CI／未驗證**。

## 4. 升降級路徑

- 同一子任務失敗兩次 → 換方法或提高 reasoning effort，並附完整失敗軌跡。
- 高 effort 也卡住 → 換 fresh-context agent、換外部來源、或走 `20-judgment.md` §3 問使用者。
- 難題被高 effort 解出模式後，把 pattern 寫進 prompt，交給較輕的 worker 批次套用。
- 同一件事最多重試兩輪。兩輪後還不行，代表方向錯了，換方法或問人。

## 5. 驗證不自驗（鐵律三）

宣稱完成之前：

- **檔案產出** → 派 `verifier` 做 read-back：給它「產出檔案路徑＋驗收條件清單」，逐條判 PASS/FAIL。
- **程式碼** → 跑測試或實跑，以指令輸出為證。沒有測試就先寫一個最小驗證腳本。
- **高風險判斷** → 加第二意見；不一致時把分歧點呈給使用者。
- 驗證 prompt 要求找碴而不是背書：「假設這份產出有問題，找出來」比「確認這份沒問題」有效。
